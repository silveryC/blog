关于我们的ClassLoader

参考：老大难的 Java ClassLoader 再不理解就老了

每个类都会标记他是由哪一个ClassLoader加载的；
所以我们可以通过 getContextClassLoader()方法获取到；
其实在JVM的众多加载其中，AppClassLoader是直接面向我们用户的加载器，它会加载 Classpath 环境变量里定义的路径中的 jar 包和目录。我们自己编写的代码以及使用的第三方 jar 包通常都是由它来加载的；
我们也可以通过 ClassLoader 类提供的静态方法 getSystemClassLoader() 得到

这就是为什么在代码中，会一句代码来对加载器进行检查： loader = (cl == null) ? ClassLoader.getSystemClassLoader() : cl;

找到配置

根据上面的分析，我在”\logback\logback-classic\src\main\resources\META-INF\services\”路径下，找到了名为org.slf4j.spi.SLF4JServiceProvider文件（文件直接以类名命名，以类名为搜索条件，这个主意真好）。
文件里面的内容是： ch.qos.logback.classic.spi.LogbackServiceProvider；

同时我也去看了看 log4j2 的配置文件，也有一个名为org.slf4j.spi.SLF4JServiceProvider的文件。
内容是：org.apache.logging.slf4j.SLF4JServiceProvider

也就是说，只要：

  1. 日志实现了org.slf4j.spi.SLF4JServiceProvider接口；
  2. 并在“META-INF\services\”路径下添加一个名为：“org.slf4j.spi.SLF4JServiceProvider”的文件
  3. 在里面添加org.slf4j.spi.SLF4JServiceProvider接口的类全路径名称（如 ch.qos.logback.classic.spi.LogbackServiceProvider）

我们就可以通过slf4j，轻松的使用不同的日志服务（只要日志服务做了以上步骤）
也可以说，Slf4j的Slf4j其实只是一个工具类，帮我们动态配置不同的日志服务，帮我们实现了组件和服务的解耦。

现在想起来，我们在项目中只是引用了slf4j和logback的包，却可以通过LoggerFactory.getLogger()轻松使用，
而不需要配置其他的（日志文件打印格式有默认的，不配置也可以的），原来是这么回事。

附录

LoggerFactory.INITIALIZATION_STATE

INITIALIZATION_STATE是用来判断初始化状态的，谁的初始化？ 成员变量PROVIDER的；

上述代码可以看到，作为LoggerFactory类的成员变量，PROVIDER在LoggerFactory初始化的时候其实是null的；
作为服务提供商PROVIDER会帮我们提供各种服务，比如给我们ILoggerFactory工厂。
往往只需要一个提供商就足够了，多的基本上是浪费（相当于一个单例)
而INITIALIZATION_STATE成员变量就是帮我们控制单例的：

  • 控制何时创建PROVIDER： 一般是在getProvider()的同步代码块里面创建
  • 控制何时销毁PROVIDER ： 当调用reset()方法的时候会被销毁，下一次创建新的提供商
  • 控制返回何种提供商：提供商拥有两种实现类 NOPServiceProvider类和SubstituteServiceProvider类，INITIALIZATION_STATE成员变量控制返回

logback源码阅读-如何阅读

发表于 2019-11-15 | 分类于 logback

前言

虽然是一名小白，但是在项目里面使用日志记录库也有很多次了；
之前都是看了一些简单如何配置日志的文章，依葫芦画瓢的配置；
但是对于原理、logback、log2j 什么的关联关系都一无所知；
所以就想着阅读一下源码。

方案

看了网上很多的源码阅读方式，总结了出来就是，你必须下载源码；
但是下载源码过后呢？从哪里开始看？怎么看？都是问题。
所以我去找了大佬咨询。。。
这里列出大佬给我的建议：

  1. 到github上去fork一份logback的源码到自己的仓库；
  2. 把自己仓库的logback源码克隆到本地；
  3. 建立demo项目关联本地源码，调试阅读logback源码，并在源码上做笔记；
  4. 因为是自己的仓库源码，笔记可以直接push到github上永久保存；
  5. 做项目使用logback的时候，关联本地源码（带有自己的笔记），这样做项目调试的时候，可以顺便巩固之前阅读的源码内容；

这个方案让我简直眼睛一亮，不仅可以用来阅读logback，还适用于所有的其他源码阅读，所以我决定尝试一下
